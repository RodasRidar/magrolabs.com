# Magrolabs Component Builder

You are scaffolding code for the **magrolabs.com** Angular 21 storefront. The project uses standalone components, signal-based reactivity, Tailwind 3 with a brand-specific theme, lazy-loaded routes, and a custom UI primitive library under `src/app/shared/ui/`.

Two kinds of things can be built with this skill — pick the right one before generating code:

| Kind | Selector prefix | Location | Purpose |
|---|---|---|---|
| **UI primitive** | `ml-` | `src/app/shared/ui/<name>/` | Reusable presentational component (button, badge, card, input, modal…) |
| **View / page** | `app-` | `src/app/<feature>/pages/<name>/` or `src/app/<feature>/<name>/` | Routed page composed of UI primitives + services |

If the user's request is ambiguous (e.g. "create a profile component"), ask whether they want a reusable UI primitive or a routed view before scaffolding.

---

## Step 0 — Always verify before generating

1. Confirm the working directory is `magrolabs.com` (check for `tailwind.config.js` with the `primary` color tokens and `angular.json`).
2. Read `tailwind.config.js` once so you have the active design tokens in memory. Never invent colors — only use the tokens listed in [Design tokens](#design-tokens) below.
3. Confirm Angular version from `package.json` is `^21.x`. If it has been upgraded, adjust to whatever the project currently uses (signal forms, new control flow, etc.).

## Step 1 — Generate the file scaffold with the Angular CLI

Always use the CLI to scaffold — never hand-roll the folder structure. Tests are generated by default.

### For a UI primitive

```bash
npx ng generate component shared/ui/<name> --selector=ml-<name> --standalone --style=css --inline-style=false
```

- Use `--inline-template` only for very small primitives (≤ ~15 lines of HTML) like `badge`, `card`, `spinner`, `page-header`. For anything larger, keep the separate `.component.html`.
- Tests are emitted by default. Do not pass `--skip-tests` unless the user explicitly asks.
- The `--style=css` flag matches the project (no SCSS, no SASS).

### For a view / page

```bash
npx ng generate component <feature>/pages/<name> --selector=app-<name> --standalone --style=css --inline-style=false
```

- `<feature>` is typically `account`, `ecommerce/landing`, `ecommerce/signup`, or `shared` (only for cross-feature pages like `not-found`).
- After generation, the component class should be exported via `export class` (or `export default class` if it is *only* lazy-loaded — see `shared/pages/not-found/not-found.component.ts` for that pattern).

After generation, **take note of the printed path** and immediately overwrite the boilerplate to match the conventions in the next steps.

## Step 2 — Rewrite the generated component to match conventions

### Shared rules (apply to both UI and views)

- **Standalone, no NgModule.** Do not import `CommonModule` unless you use a pipe (`async`, `currency`, etc.). Modern control flow (`@if`, `@for`, `@switch`) is built into the template and does not need `CommonModule`.
- **Signal inputs/outputs only.** Use `input.required<T>()`, `input<T>(default)`, `model<T>()`, `output<T>()`. Never use the legacy `@Input()` / `@Output()` decorators in new code.
- **State.** Use `signal()` for mutable state and `computed()` for derived state. Inject services with the `inject()` function — never constructor injection.
- **Templates.** Use `@if` / `@for` / `@switch`. For `@for`, always include `track`. No `*ngIf` or `*ngFor`.
- **Class composition.** When a component has variants (size, color, type), expose them as a typed input and compute the final class string with `computed()` + a `Record<Variant, string>` map. See [The variant pattern](#the-variant-pattern) below.
- **Host class.** Add `host: { class: 'block' }` on UI primitives so they participate in flex/grid layouts correctly. Pages do not need this.
- **No emojis in code** unless the user explicitly asks (some existing copy contains them, but new code should not introduce more).

### UI primitive specifics

- Selector must be `ml-<kebab-name>`.
- File lives in `src/app/shared/ui/<name>/`.
- Read [references/ui-primitive-pattern.md](references/ui-primitive-pattern.md) for the full template and a worked example.
- If the primitive is a form control (input, select, textarea, checkbox, radio, password input, etc.), it MUST implement `ControlValueAccessor` with the `NG_VALUE_ACCESSOR` provider via `forwardRef`. See [references/form-control-pattern.md](references/form-control-pattern.md).

### View / page specifics

- Selector must be `app-<kebab-name>`.
- File lives in `src/app/<feature>/pages/<name>/` (or `src/app/<feature>/<name>/` for top-level feature pages).
- Configure SEO inside `ngOnInit` via the injected `SeoService` — set title, description, and `setIndexFollow(false)` for any authenticated/private page.
- Wrap content in the standard page shell: `<ml-page-header [title]="...">` then `<main><div class="max-w-7xl mx-auto sm:px-6 lg:px-8"><div class="px-4 py-8 sm:px-0">…</div></div></main>`.
- Compose with existing UI primitives (`ml-card`, `ml-button`, `ml-action-card`, `ml-badge`, `ml-alert`, `ml-input`, `ml-select`, `ml-loyalty-meter`, `ml-page-header`, etc.) instead of redefining styled wrappers.
- Show a skeleton state while data loads (gray `animate-pulse` blocks inside `ml-card`s on the same grid as the real content). See `src/app/account/pages/cuenta/cuenta.component.html` for the canonical skeleton.
- Clean up RxJS subscriptions with `takeUntilDestroyed()` (preferred for new code) or the existing `Subject<void>` + `destroyRef.onDestroy()` pattern if matching surrounding code.
- Read [references/view-pattern.md](references/view-pattern.md) for the full anatomy.

## Step 3 — Wire the view into routing (views only)

Edit `src/app/app.routes.ts` and add a lazy-loaded route. Always use `loadComponent`, never eager imports:

```ts
{
  path: '<url-slug>',
  loadComponent: () =>
    import('./<feature>/pages/<name>/<name>.component').then(m => m.<Name>Component),
  canActivate: [refLinkGuard], // or [authGuard] for /cuenta children, etc.
}
```

Guard selection:
- Public marketing/storefront page → `refLinkGuard`.
- Authenticated user area → `authGuard` (typically as a child of the `cuenta` route).
- Auth-only-when-logged-out (login, signup) → `notAuthGuard` + `refLinkGuard`.

If the page belongs under an existing parent (e.g. `/cuenta/...`, `/registro/...`, `/atencion-cliente/...`), nest it inside the parent's `children` array rather than adding a new top-level route.

## Step 4 — Update the generated `.spec.ts`

The CLI emits a minimal "should create" test. Keep it, but:

- For UI primitives with required inputs, set those inputs via `componentRef.setInput('name', value)` before `detectChanges()` — otherwise the test crashes on `input.required`.
- For form-control primitives (ControlValueAccessor), test through a `TestHostComponent` with a `FormControl`, as in `src/app/shared/ui/input/input.component.spec.ts`. That is the canonical example — read it before writing a new form control spec.
- For variant primitives, add at least one test per variant verifying the expected class is on the rendered element.

Tests use Jasmine + Karma (the project's current setup), not Vitest. Use `expect(...).toBeTrue()` / `toBeFalse()` (Jasmine matchers), `By.css(...)` from `@angular/platform-browser`, and `fixture.detectChanges()`.

## Design tokens

These are the only colors/fonts you may reference. They come from `tailwind.config.js` — do not invent new ones, do not hard-code hex values, and do not introduce arbitrary Tailwind colors like `text-orange-500` when a brand token fits.

**Brand**
- `primary` (`#d07d66`) — main CTA, active links, primary buttons
- `primary-hover` (`#d7816a`) — hover on primary
- `primary-dark` (`#b86b56`) — hover on darker variants
- `primary-muted` (`#d89785`) — focus rings on form controls
- `primary-surface` (`#fae8e3`) — badges, chips, soft backgrounds
- `primary-surface-light` (`#ffe8e2`) — nav active background

**Neutrals**
- `surface` (`#f6f6f6`) — cards, sections (default `ml-card` background)
- `surface-alt` (`#f5f5f0`) — alternative surface

**Accent**
- `accent` (`#6f67eb`) — spinners, secondary indicators

**Typography**
- `font-sans` resolves to `SF Pro Display` then Apple/system fallbacks. Do not declare an alternative font on new components.

**Status colors** (for badges, alerts, semantic UI) — Tailwind defaults:
- success → `green-{50,100,400,500,800}`
- warning → `yellow-{50,100,400,500,800}`
- error → `red-{50,100,400,500,800}`
- info → `blue-{50,100,400,500,800}`

The dark gradient header used in hero strips and accent banners is `bg-gradient-to-r from-[#1f1b19] to-primary-hover` — that one hard-coded hex is intentional and already used in `ml-action-card` and the dashboard banner. You may reuse it; do not introduce other arbitrary hex values.

## The variant pattern

This is the project's single most repeated pattern. Memorize it.

```ts
import { Component, computed, input } from '@angular/core';

export type BadgeColor = 'green' | 'yellow' | 'red' | 'blue' | 'indigo' | 'gray';
export type BadgeSize  = 'sm' | 'md';

@Component({
  selector: 'ml-badge',
  standalone: true,
  template: `<span [class]="badgeClass()">{{ label() }}</span>`,
})
export class BadgeComponent {
  color = input.required<BadgeColor>();
  label = input.required<string>();
  size  = input<BadgeSize>('sm');

  protected readonly badgeClass = computed(() => {
    const colorMap: Record<BadgeColor, string> = {
      green:  'bg-green-100 text-green-800',
      yellow: 'bg-yellow-100 text-yellow-800',
      red:    'bg-red-100 text-red-800',
      blue:   'bg-blue-100 text-blue-800',
      indigo: 'bg-indigo-100 text-indigo-800',
      gray:   'bg-gray-200 text-gray-800',
    };
    const sizeMap: Record<BadgeSize, string> = {
      sm: 'px-2.5 py-0.5',
      md: 'px-3 py-1',
    };
    return `inline-flex items-center text-xs font-medium rounded-full ${sizeMap[this.size()]} ${colorMap[this.color()]}`;
  });
}
```

Key points:
- `Record<Variant, string>` ensures every variant has a class string (TS catches missing cases).
- `protected readonly` on computed signals exposed only to the template.
- Tailwind classes are written as full strings — never assembled from fragments like `` `text-${color}-800` `` because Tailwind's JIT can't see them.
